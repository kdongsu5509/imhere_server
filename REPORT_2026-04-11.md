## 1. 작업 개요

오늘 세션에서는 ImHere 백엔드 서버의 운영 배포 준비를 위한 전반적인 인프라 및 자동화 작업을 수행하였다. 크게 네 가지 영역으로 나뉜다.

1. **GitHub Actions CI/CD 파이프라인 구축** — 테스트 자동화(CI)와 AWS ECR 기반 도커 이미지 빌드·배포 자동화(CD)
2. **Prometheus + Grafana 모니터링 스택 구성** — 메트릭 수집·시각화 설정
3. **docker-compose 다중 파일 분리 및 AWS 의존성 제거** — 환경 중립적 구성으로 전환
4. **인프라 아키텍처 설계 검토** — 100 TPS 기준 최적 배포 구성 및 비용 분석

## 2. CI 파이프라인 구축

### 2.1 파일 위치

```
.github/workflows/ci.yml
```

### 2.2 트리거 전략

CI 워크플로우는 모든 브랜치 푸시와 `main` 브랜치로의 Pull Request에서 실행된다.

```yaml
on:
  push:
    branches:
      - "**"
  pull_request:
    branches:
      - main
```

모든 브랜치에서 테스트가 실행되도록 구성하여, 기능 브랜치에서 이미 결함을 발견할 수 있도록 하였다. PR 시점에서만 테스트를 실행하면 피드백 주기가 길어지므로 이 방식을 채택하였다.

### 2.3 실행 환경

- **Runner**: `ubuntu-latest` (GitHub-hosted)
- **JDK**: Amazon Corretto 21 (`actions/setup-java@v4`)
- **Gradle 캐시**: `~/.gradle/caches`, `~/.gradle/wrapper`를 `actions/cache@v4`로 캐싱

Gradle 캐시 키는 OS + Gradle 파일 해시로 구성하여 의존성 변경 시에만 캐시가 무효화된다.

```yaml
key: gradle-${{ runner.os }}-${{ hashFiles('**/*.gradle', 'gradle/wrapper/gradle-wrapper.properties') }}
```

### 2.4 Testcontainers 처리

이 프로젝트는 Redis와 RabbitMQ를 Testcontainers로 관리한다.

```kotlin
// TestRedisContainer.kt
val redisContainer = GenericContainer(DockerImageName.parse("redis:alpine"))
    .withExposedPorts(REDIS_PORT)
    .apply { start() }

// TestRabbitMQContainer.kt
val rabbitMqContainer = GenericContainer(DockerImageName.parse("rabbitmq:4.3.0-rc.0-management-alpine"))
    .withExposedPorts(RABBIT_MQ_PORT)
    .apply { start() }
```

`@DynamicPropertySource`를 통해 컨테이너가 바인딩된 동적 포트를 Spring 프로퍼티에 자동으로 주입한다. GitHub Actions의 `ubuntu-latest` 러너는 Docker 데몬이 기본 포함되어 있으므로 별도의 서비스 컨테이너 설정 없이 Testcontainers가 정상 동작한다.

Ryuk(컨테이너 정리 프로세스) 비활성화 설정을 추가하여 CI 환경에서 발생할 수 있는 Ryuk 관련 타임아웃 오류를 방지하였다.

```yaml
env:
  TESTCONTAINERS_RYUK_DISABLED: "true"
```

### 2.5 아티팩트 업로드

테스트 성공/실패 여부와 무관하게(`if: always()`) 결과물을 아티팩트로 업로드하여 실패 원인 분석이 가능하도록 하였다.

- **test-results**: `build/reports/tests/test/` — JUnit XML + HTML 리포트
- **jacoco-report**: `build/jacocoHtml/` — 커버리지 리포트

## 3. CD 파이프라인 구축

### 3.1 파일 위치

```
.github/workflows/cd.yml
```

### 3.2 트리거 전략 — workflow_run

CD는 CI가 `main` 브랜치에서 성공적으로 완료된 경우에만 실행된다.

```yaml
on:
  workflow_run:
    workflows: ["CI"]
    branches:
      - main
    types:
      - completed
```

```yaml
if: ${{ github.event.workflow_run.conclusion == 'success' }}
```

`workflow_run` 트리거를 사용하면 CI와 CD를 완전히 분리된 파일로 관리하면서도, CI 성공을 전제 조건으로 강제할 수 있다. `push` 이벤트로 CD를 직접 트리거하면 테스트 없이도 배포가 실행될 수 있으므로 이 방식을 채택하였다.

### 3.3 빌드 전략 — GHA에서 JAR 빌드 후 경량 이미지 패키징

기존 Dockerfile은 멀티스테이지 빌드 방식으로, Docker 빌드 레이어 안에서 `./gradlew build`를 실행하였다. 이 방식의 문제점은 다음과 같다.

- Gradle 의존성 캐시가 GHA의 `actions/cache`와 연동되지 않아 매 빌드마다 수백 MB를 재다운로드
- 빌드 컨텍스트 전체를 Docker 데몬에 전달하므로 느림
- `COPY del src` 오타가 있는 기존 `Dockerfile`은 실제 동작하지 않음

변경된 전략은 다음과 같다.

```
1. GHA에서 Gradle 캐시를 활용하여 ./gradlew bootJar -x test 실행 (~30-60초)
2. Dockerfile.release를 사용하여 빌드된 JAR만 복사하는 경량 런타임 이미지 생성
3. ECR에 SHA 태그와 latest 태그로 푸시
```

#### Dockerfile.release (신규 생성)

```dockerfile
FROM amazoncorretto:21-alpine-jdk
WORKDIR /app
COPY build/libs/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

빌드 레이어가 없으므로 이미지 크기가 작고, 레이어 캐싱 효율이 높다.

### 3.4 AWS ECR 연동

```yaml
- name: Configure AWS credentials
  uses: aws-actions/configure-aws-credentials@v4
  with:
    aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
    aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
    aws-region: ap-northeast-2

- name: Login to Amazon ECR
  uses: aws-actions/amazon-ecr-login@v2
```

이미지는 커밋 SHA 태그와 `latest` 태그 두 가지로 푸시한다. SHA 태그는 특정 배포 버전으로 롤백 시 사용하고, `latest`는 EC2에서 `docker compose pull`로 항상 최신 버전을 가져올 수 있도록 한다.

```bash
docker build -f Dockerfile.release \
  -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG \
  -t $ECR_REGISTRY/$ECR_REPOSITORY:latest \
  .
docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG
docker push $ECR_REGISTRY/$ECR_REPOSITORY:latest
```

### 3.5 EC2 배포 — appleboy/ssh-action

`appleboy/ssh-action@v1.2.0`을 사용하여 EC2에 SSH로 접속 후 배포한다.

```yaml
- name: Deploy to EC2 via SSH
  uses: appleboy/ssh-action@v1.2.0
  with:
    host: ${{ secrets.EC2_HOST }}
    username: ${{ secrets.EC2_USER }}
    key: ${{ secrets.EC2_SSH_PRIVATE_KEY }}
    script: |
      aws ecr get-login-password --region ap-northeast-2 \
        | docker login --username AWS --password-stdin 518033442106.dkr.ecr.ap-northeast-2.amazonaws.com
      cd ${{ secrets.EC2_DEPLOY_PATH }}
      docker compose pull dsko
      docker compose up -d --no-deps dsko
      docker image prune -f
```

`--no-deps` 플래그는 `dsko`(앱) 컨테이너만 재시작하고 Redis, Prometheus, Grafana에는 영향을 주지 않도록 한다. 배포 후 미사용 이미지를 자동 정리하여 디스크 낭비를 방지한다.

ECR 로그인은 EC2 인스턴스 IAM 역할(AmazonEC2ContainerRegistryReadOnly)을 사용하므로, EC2에 AWS 자격증명을 별도로 저장할 필요가 없다.

### 3.6 필요한 GitHub Secrets 목록

| Secret | 용도 |
|---|---|
| `AWS_ACCESS_KEY_ID` | ECR 푸시용 IAM 액세스 키 |
| `AWS_SECRET_ACCESS_KEY` | ECR 푸시용 IAM 시크릿 키 |
| `EC2_HOST` | EC2 퍼블릭 IP 또는 도메인 |
| `EC2_USER` | SSH 유저명 (ubuntu / ec2-user) |
| `EC2_SSH_PRIVATE_KEY` | PEM 키 파일 내용 전체 |
| `EC2_DEPLOY_PATH` | docker-compose.yml 위치 경로 |

## 4. Prometheus + Grafana 모니터링 스택

### 4.1 설정 파일 구조

```
prometheus.yml
grafana/
└── provisioning/
    ├── datasources/
    │   └── prometheus.yaml
    └── dashboards/
        ├── provider.yaml
        └── imhere.json
```

### 4.2 prometheus.yml

```yaml
global:
  scrape_interval: 30s
  evaluation_interval: 30s

scrape_configs:
  - job_name: "imhere"
    metrics_path: "/xCJR4cpW3VSxH8esujE2maQUJaaF6fSsPVBmMgAIcDmW9VUW9wvrrUP6jq8gF8aSVuyKfE7fEvSzkI842FH3NhBFG3UN5Zns8Vw6rz75h7dRgb2R6EMBVT6yMG8a5WDb/prometheus"
    static_configs:
      - targets: [ "dsko:4861" ]
```

`application-monitoring.yaml`에 설정된 관리 포트(4861)와 난독화된 base-path를 그대로 사용한다. 메트릭 엔드포인트가 외부에 노출되지 않도록 포트 4861은 EC2 보안 그룹에서 인바운드를 차단하고, Docker 내부 네트워크(app-network)에서만 접근한다. 스크랩 간격은 사용자가 15s → 30s로 조정하였다.

### 4.3 Grafana Provisioning

Grafana는 컨테이너 기동 시 `provisioning/` 디렉토리를 읽어 데이터소스와 대시보드를 자동 등록한다. 별도의 수동 설정 없이 즉시 사용 가능한 상태로 기동된다.

**datasources/prometheus.yaml**: Prometheus를 기본 데이터소스로 자동 등록  
**dashboards/provider.yaml**: `/var/lib/grafana/dashboards`를 대시보드 파일 경로로 설정  
**dashboards/imhere.json**: Spring Boot Micrometer 기반 커스텀 대시보드

### 4.4 imhere.json 대시보드 구성

대시보드는 다음 섹션으로 구성된다.

**HTTP 섹션**
- HTTP Request Rate by Status — 상태 코드별 초당 요청 수 (reqps)
- HTTP Latency p95/p99 — URI별 응답 지연 분위수 (histogram_quantile)
- 5xx Error Rate — 5분 평균 에러율 (stat 패널, 1 이상 시 빨간색)
- Total Request Rate — 전체 요청률 (stat 패널)

**JVM 섹션**
- JVM Heap Memory — 힙 used/max (bytes)
- JVM Threads — live/daemon/peak 스레드 수
- GC Pause Rate — GC 일시정지 시간 비율 (action/cause별)
- CPU Usage — 프로세스/시스템 CPU 사용률

**HikariCP 섹션**
- HikariCP Connections — active/idle/max/pending 커넥션 수
- HikariCP Acquire/Usage Latency — 커넥션 획득·사용 p99 지연

대시보드 자동 갱신 주기는 30초, 기본 조회 범위는 최근 1시간이다.

### 4.5 보안 설정

Prometheus와 Grafana 포트를 `127.0.0.1`에만 바인딩하여 외부 직접 접근을 차단하였다.

```yaml
prometheus:
  ports:
    - "127.0.0.1:9090:9090"

grafana:
  ports:
    - "127.0.0.1:3000:3000"
```

로컬 접근 시 SSH 터널을 사용한다.

```bash
ssh -L 3000:localhost:3000 -L 9090:localhost:9090 ec2-user@<EC2_HOST>
```

## 5. docker-compose 파일 분리 및 AWS 의존성 제거

### 5.1 변경 전 문제점

기존 `docker-compose.yml`은 다음 문제를 가지고 있었다.

1. **AWS ECR 이미지 URL 하드코딩** — `518033442106.dkr.ecr.ap-northeast-2.amazonaws.com/iamhere:latest`
   로컬 개발이나 다른 환경에서 실행 불가
2. **모든 서비스가 단일 파일에 혼재** — 배포 서버 구성이 달라질 경우 파일 분리 불가
3. **PostgreSQL, RabbitMQ 서비스 누락** — 실제 운영에 필요한 서비스가 정의되지 않음
4. **환경변수 미사용** — 민감 정보(비밀번호 등)가 하드코딩될 위험

### 5.2 변경 후 파일 구조

```
docker-compose.yml        — Server 1: App + Prometheus + Grafana
docker-compose.infra.yml  — Server 2: Redis + RabbitMQ
docker-compose.db.yml     — Server 3: PostgreSQL
```

### 5.3 docker-compose.yml (Server 1)

앱 이미지를 `iamhere:latest`로 변경하여 레지스트리 의존성을 제거하였다. 외부 서비스(Redis, RabbitMQ, PostgreSQL)의 호스트 정보는 모두 환경변수로 주입받는다.

```yaml
app:
  image: iamhere:latest
  environment:
    - SPRING_DATA_REDIS_HOST=${INFRA_HOST}
    - SPRING_RABBITMQ_HOST=${INFRA_HOST}
    - SPRING_DATASOURCE_URL=jdbc:postgresql://${DB_HOST}:5432/${DB_NAME}
    - SPRING_DATASOURCE_USERNAME=${DB_USER}
    - SPRING_DATASOURCE_PASSWORD=${DB_PASSWORD}
```

실행 전 `.env` 파일 또는 `export` 명령으로 환경변수를 설정해야 한다.

### 5.4 docker-compose.infra.yml (Server 2)

Redis에 비밀번호 인증을 추가하고, RabbitMQ 관리 UI는 외부 노출 없이 로컬(127.0.0.1)에만 바인딩하였다.

```yaml
redis:
  command: redis-server --requirepass ${REDIS_PASSWORD}

rabbitmq:
  ports:
    - "5672:5672"
    - "127.0.0.1:15672:15672"  # 관리 UI 로컬만
  environment:
    - RABBITMQ_DEFAULT_VHOST=/main
```

### 5.5 docker-compose.db.yml (Server 3)

PostgreSQL 단독 서버. 데이터는 named volume으로 영속화하며, 포트는 앱 서버에서만 접근 가능하도록 보안 그룹으로 제한한다.

## 6. 아키텍처 설계 검토

### 6.1 단일 서버 구성의 문제점 검토

초기 구성은 모든 서비스(앱, Redis, RabbitMQ, PostgreSQL, Prometheus, Grafana)를 단일 EC2에서 Docker로 실행하는 방안이었다. 이에 대한 검토 결과 두 가지 주요 문제가 확인되었다.

**보안 문제**
- Prometheus 포트(9090) 공개 시 내부 메트릭 전체가 외부에 노출
- Grafana 포트(3000) 공개 시 인증 우회 공격 대상이 됨
- 관리 포트(4861)는 난독화 경로로 일부 보호되지만 단일 방어선

**리소스 문제**
- t3.small(2GB): Docker 데몬 + App JVM(1GB) + Redis + RabbitMQ + PostgreSQL + Prometheus + Grafana 합산 시 OOM 위험
- 특히 PostgreSQL과 Prometheus는 각각 수백 MB를 상시 점유

### 6.2 관리형 서비스 검토

상태를 가진 서비스(stateful)를 AWS 관리형으로 전환하는 방안을 검토하였다.

| 서비스 | 자체 Docker | AWS 관리형 | 차이 |
|---|---|---|---|
| PostgreSQL | EBS 볼륨 직접 관리 | RDS (자동 백업, 패치) | +$13/월 |
| Redis | 직접 운영 | ElastiCache (고가용성) | +$12/월 |
| RabbitMQ | Docker 경량 | Amazon MQ | +$15/월 |

데이터 유실 리스크와 운영 부담을 고려하면, 월 $25~40의 추가 비용은 합리적인 보험료로 판단된다.

### 6.3 100 TPS 기준 최종 아키텍처 선택

최대 TPS 100 기준으로 최적 구성을 분석하였다.

**부하 분석**
- 평균 응답시간 50ms 가정 → 동시 처리 요청 수 = 100 × 0.05 = 5개
- HikariCP 기본 풀 10 커넥션으로 DB 병목 없음
- Redis는 단일 스레드지만 초당 수만 ops 처리 가능 → cache.t4g.micro 충분
- RabbitMQ는 MQ 처리가 비동기이므로 직접적인 TPS 영향 없음

**선택한 구성**

```
EC2 t3.medium (2vCPU / 4GB)
  ├── Spring Boot App  [JVM heap ~1GB]
  ├── RabbitMQ (Docker)  [~300MB]
  └── Prometheus (Docker)  [~200MB]
      ↓ (내부 네트워크)
  ElastiCache cache.t4g.micro  [Redis, 관리형]
  RDS db.t4g.micro  [PostgreSQL, 관리형]

모니터링 시각화: Grafana Cloud 무료 티어
```

**메모리 레이아웃 (t3.medium 4GB)**

```
App JVM heap       1.0 GB
RabbitMQ           0.3 GB
Prometheus         0.2 GB
OS + Docker 데몬   0.5 GB
여유 버퍼          2.0 GB  ← 스파이크 대응 및 JVM 오버헤드
```

t3.small(2GB) 대비 t3.medium(4GB)을 선택한 이유는 트래픽 스파이크 시 GC 압박과 OOM을 방지하기 위한 여유 확보이다.

**Grafana Cloud 선택 이유**
- 무료 티어: 10,000 active series, 50GB 로그, 14일 보존
- 이 규모에서 무료 한도 내 운영 가능
- EC2에서 Grafana를 제거하면 메모리 ~300MB 확보
- 외부 접근 가능한 대시보드 URL 제공 (SSH 터널 불필요)

### 6.4 스케일 아웃 전략

100 TPS를 초과하는 시점에서의 확장 전략은 다음과 같다.

```
현재: EC2 단일 인스턴스
  ↓ TPS 증가
ALB (Application Load Balancer) 추가
  ↓
EC2 Auto Scaling Group (앱 레이어만 수평 확장)
  ├── Instance 1 (App)
  ├── Instance 2 (App)
  └── Instance N (App)
  ↓
RDS, ElastiCache, Amazon MQ는 변경 없음 (이미 분리되어 있음)
```

RabbitMQ가 단일 EC2에 있다면, TPS 급증 시 Amazon MQ로 마이그레이션이 필요하다. 이는 RabbitMQ Docker → Amazon MQ 전환만으로 가능하며, 앱 코드 변경은 불필요하다.

### 8.4 애플리케이션 설정

```bash
RABBITMQ_PASSWORD=<rabbitmq_password>

GRAFANA_PASSWORD=<grafana_password>
```

### 8.5 Dockerfile 오타 수정 필요

기존 `Dockerfile` 7번째 줄:
```dockerfile
COPY del src   # 오타 — del 폴더를 복사하려 함
```
→ 수정 필요:
```dockerfile
COPY src src
```

CD 파이프라인은 `Dockerfile.release`를 사용하므로 당장 배포에는 영향 없지만, 로컬 빌드 시 사용하는 `Dockerfile`은 수정이 필요하다.

## 10. 향후 고려사항

### 10.1 단기 (배포 직후)

- **Grafana Cloud 연동**: Prometheus 원격 쓰기(remote_write)를 설정하여 Grafana Cloud로 메트릭 전송
- **RDS 자동 백업 주기 설정**: 기본 7일 → 30일로 조정 권장
- **알림 규칙 추가**: Grafana/Alertmanager에서 5xx 급증, JVM OOM 임박(heap > 85%), DB 커넥션 고갈 시 Discord 알림 연동
- **application-secret.yaml EC2 배포 자동화**: 현재 수동으로 EC2에 올려야 함 — AWS Secrets Manager 또는 Parameter Store 연동 고려

### 10.2 중기

- **ALB 도입**: 100 TPS 초과 시 수평 확장을 위한 로드 밸런서 선행 도입
- **HTTPS 설정**: ACM 인증서 + ALB 또는 Let's Encrypt + nginx
- **Zero-downtime 배포**: 현재 `docker compose up -d`는 순간 다운타임 발생 — Blue/Green 또는 Rolling 전략 도입
- **Amazon MQ 마이그레이션**: 트래픽 증가 시 RabbitMQ를 관리형으로 전환

### 10.3 장기

- **EKS/ECS 전환**: 수백 TPS 이상에서 컨테이너 오케스트레이션 도입
- **Observability 강화**: 분산 트레이싱(OpenTelemetry + Jaeger/Tempo) 도입 — 현재 MDC traceId 기반이므로 자연스러운 확장 가능
